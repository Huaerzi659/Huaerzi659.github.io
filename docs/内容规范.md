# Waltz, Then Think 内容规范

这份文档用来帮助你稳定地写文章、改文章和检查文章。它不是文学规则，而是一套维护网站时不会迷路的约定。

## 1. 写文章的固定流程

1. 先想清楚文章属于哪一类：技术教程、经验复盘、感想随笔。
2. 在 `src/content/posts/` 新建 Markdown 文件。
3. 文件名使用小写英文短横线，例如 `from-zero-to-blog.md`。
4. 从 `docs/templates/post-template.md` 复制 frontmatter。
5. 写完正文后运行 `npm run verify:content`。
6. 再运行常规验证和构建。

## 2. 文件命名

文章文件名会进入网址，所以发布后不要随便改。

正确：

```text
stable-internet-access.md
from-zero-to-blog.md
```

不推荐：

```text
我的文章.md
stable_internet_access.md
StableInternet.md
```

## 3. Frontmatter

每篇文章开头都要有：

```yaml
---
title: "文章标题"
description: "一句清楚的文章摘要"
date: "2026-06-12"
tags: ["技术"]
featured: false
---
```

`title` 是页面标题。正文里不要再写 `# 一级标题`。

`description` 是文章摘要，会用于文章卡片、搜索引擎和 RSS。它要说明读者能从文章里得到什么。

`date` 使用 `YYYY-MM-DD`。

`tags` 建议 1 到 3 个。

`featured` 控制是否出现在首页精选区域。

## 4. 标签池

当前推荐标签：

```text
技术
经验
感想
灵感
工具
复盘
```

新增标签前先想：以后会不会继续写这个主题。如果只会出现一次，优先用已有标签。

## 5. Markdown 标题

文章页面已经自动生成 `h1`，正文从 `##` 开始。

```md
## 第一节

### 第一节里的小节

#### 更细的小点
```

不要这样写：

```md
# 我的文章标题
```

标题不要跳级。比如有了 `##`，下面可以接 `###`；不要直接从 `##` 跳到 `####`。

## 6. 段落和列表

段落尽量短一点，尤其要考虑手机阅读。

建议：

- 一个段落讲一件事。
- 很长的段落拆成 2 到 3 段。
- 列表用于并列信息，不要把整篇文章写成列表。
- 列表前后留空行，让 Markdown 更稳定。

## 7. 图片

图片放在：

```text
public/images/articles/<文章文件名去掉 .md>/
```

引用方式：

```md
![第一次配置流程图](/images/articles/stable-internet-access/setup-flow.svg)
```

图片 alt 文本要写清楚，不要空着。

图片文件名也用小写英文短横线，例如：

```text
setup-flow.svg
cover-stable-internet-guide.png
```

## 8. 技术教程结构

技术教程适合这种结构：

```md
开头：说明适合谁、解决什么问题

## 先理解概念

## 准备工作

## 操作步骤

## 常见问题

## 总结

## 参考链接
```

技术文章最重要的是“读者能跟着做”。不要只贴命令，也要写清楚目的和结果。

## 9. 经验复盘结构

经验复盘适合这种结构：

```md
开头：发生了什么

## 背景

## 我做了什么

## 哪里卡住了

## 解决方式

## 下次怎么做
```

复盘文章最有价值的部分通常不是“我成功了”，而是“我怎么判断问题在哪里”。

## 10. 感想随笔结构

感想随笔可以松一点，但最好仍然有起承转合：

```text
开头：一个真实触发点
中段：展开想法
后段：回到自己的判断
```

随笔不需要硬凑小标题。如果文字短、节奏完整，可以只写段落。

## 11. 发布前检查

先检查内容：

```powershell
npm run verify:content
```

再检查已有站点规则：

```powershell
npm run verify:ux
npm run verify:motion
npm run verify:rss
```

构建使用项目已知可用 Node：

```powershell
& 'C:\Users\刘\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\bin\node.exe' .\node_modules\astro\bin\astro.mjs build
```

## 12. 常见错误

如果 `verify:content` 提示 `body should not contain # headings`，说明正文里用了一级标题。把它改成 `##`。

如果提示图片不存在，检查路径是不是从 `/images/articles/` 开始，以及图片是否真的在 `public/images/articles/` 里。

如果提示标签未知，先从标签池里选择一个已有标签。

如果提示代码围栏没有成对，检查文章里是否有单独一个未闭合的：

````md
```js
console.log("hello");
```
````

## 13. 最小发布清单

发文章前至少确认：

```text
文件名是英文短横线
frontmatter 五个字段都在
description 读起来像摘要
tags 没超过 3 个
正文没有 # 一级标题
图片有 alt 文本
npm run verify:content 通过
Astro build 通过
```

这些都通过以后，再考虑提交和上线。
