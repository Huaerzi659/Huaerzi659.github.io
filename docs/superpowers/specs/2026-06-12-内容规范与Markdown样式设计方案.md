# 内容规范与 Markdown 样式设计方案

## 背景

Waltz, Then Think 已经具备基础博客能力：Astro 页面、Markdown 文章、RSS、主题索引页、双风格主题切换和文章详情页。

目前内容层面的主要问题是：

- 文章 frontmatter 只由 Astro schema 做基础校验，缺少适合博客长期维护的写作规则。
- 新文章没有模板，容易在标题、摘要、标签、图片路径、正文结构上每次重新想。
- Markdown 正文样式覆盖不完整，长文里的图片、表格、引用、代码块、分割线、标题层级还不够像正式文章。
- 内容质量检查依赖人工判断，不利于站主作为网页新手独立维护。

这次目标是把“内容规范”做成可以长期复用的维护系统，而不是只写一段说明。

## 目标

本次实现完整档内容规范，包含三层：

1. 写作规范：让 Waltz 知道每篇文章应该怎么命名、怎么写 frontmatter、怎么组织 Markdown。
2. 自动检查：让项目能在本地用脚本检查常见内容问题，减少靠感觉维护。
3. 正文样式：让 Markdown 渲染效果更完整，尤其是技术教程和长文的阅读体验。

成功标准：

- 新增 `docs/内容规范.md`，用中文解释日常写作规则。
- 新增 `docs/templates/post-template.md`，作为新文章模板。
- 新增 `scripts/verify-content.mjs` 和 `npm run verify:content`。
- `verify:content` 能检查现有文章并通过。
- `.prose` 样式覆盖标题、段落、链接、列表、引用、图片、表格、代码块、分割线。
- `cyber` 和 `anime-scroll` 两个主题下的正文样式都协调，anime-scroll 不残留不合适的霓虹光影。

## 非目标

本次不做这些功能：

- 不引入运行时依赖或新的 Markdown 插件。
- 不做评论系统。
- 不做站内搜索。
- 不做后台 CMS。
- 不批量重写现有文章内容，只做必要的小幅规范化修正。
- 不改变公开站点默认主题。

## 内容规范范围

### 文件命名

文章文件继续放在：

```text
src/content/posts/
```

文件名规范：

```text
小写英文
单词之间用短横线
以 .md 结尾
不使用空格、中文、下划线
```

示例：

```text
stable-internet-access.md
from-zero-to-blog.md
```

文件名会成为文章 URL 的一部分，所以保持稳定。文章发布后不随便改文件名，避免旧链接失效。

### Frontmatter

每篇文章必须包含：

```yaml
---
title: "文章标题"
description: "文章摘要"
date: "2026-06-12"
tags: ["技术"]
featured: false
---
```

规则：

- `title`：必填，建议 6 到 28 个中文字符，不要和正文一级标题重复。
- `description`：必填，建议 40 到 100 个中文字符，用一句话说明读者能获得什么。
- `date`：必填，格式为 `YYYY-MM-DD`。
- `tags`：必填，建议 1 到 3 个。
- `featured`：必填，布尔值，只能是 `true` 或 `false`。

### 标签规范

保留短标签，避免同义词泛滥。

推荐标签池：

```text
技术
经验
感想
灵感
工具
复盘
```

允许以后扩展，但每次新增标签都应该有明确用途。

不推荐同时出现这些近义混用：

```text
技术 / 编程 / tech
感想 / 随笔 / 碎碎念
经验 / 记录 / 日志
```

### Markdown 标题层级

文章页面已经用 frontmatter 的 `title` 渲染 `<h1>`，所以正文里不要再写 `# 一级标题`。

正文标题从 `##` 开始：

```md
## 一级正文段落

### 二级正文段落

#### 更细的小节
```

规则：

- 不跳级，例如不要从 `##` 直接跳到 `####`。
- 短随笔可以没有小标题。
- 长教程建议至少有 3 个 `##` 小节。

### 段落和列表

段落保持短一点，适合手机阅读。

建议：

- 一个段落讲一件事。
- 长段落尽量拆成 2 到 3 段。
- 列表用于并列信息，不把整篇文章写成列表。
- 列表项前后保持空行，让 Markdown 更稳定。

### 图片

文章图片放在：

```text
public/images/articles/<文章 slug>/
```

图片引用示例：

```md
![第一次配置流程图](/images/articles/stable-internet-access/setup-flow.svg)
```

规则：

- 图片必须有 alt 文本。
- 本地图片路径必须从 `/images/articles/` 开始。
- 图片文件名使用小写英文短横线。
- 封面图建议命名为 `cover-<slug>.svg`、`cover-<slug>.png` 或 `cover-<slug>.webp`。

### 技术教程结构

技术教程推荐结构：

```md
开头：说明适合谁、解决什么问题

## 先理解概念

## 准备工作

## 操作步骤

## 常见问题

## 总结

## 参考链接
```

### 经验复盘结构

经验复盘推荐结构：

```md
开头：发生了什么

## 背景

## 我做了什么

## 哪里卡住了

## 解决方式

## 下次怎么做
```

### 感想随笔结构

感想随笔不需要过度模板化，但要保留清晰起承转合：

```md
开头：一个真实触发点
中段：展开想法
后段：回到自己得到的判断
```

## 自动检查设计

新增：

```text
scripts/verify-content.mjs
```

新增 package script：

```json
"verify:content": "node scripts/verify-content.mjs"
```

脚本使用 Node 原生能力，不新增依赖。

检查项：

- `src/content/posts/` 至少有一篇 Markdown。
- 文件名匹配 `^[a-z0-9]+(?:-[a-z0-9]+)*\.md$`。
- 每篇文章有 frontmatter。
- frontmatter 包含 `title`、`description`、`date`、`tags`、`featured`。
- `date` 符合 `YYYY-MM-DD`。
- `featured` 是 `true` 或 `false`。
- `tags` 是 YAML 行内数组写法，例如 `["技术", "经验"]`。
- 标签数量为 1 到 3 个。
- 标签来自推荐标签池。
- `description` 长度在 20 到 120 个字符之间，脚本先采用较宽范围，避免误伤现有文章。
- 正文不包含 `# 一级标题`。
- 正文标题不跳级。
- 图片语法必须有 alt 文本。
- 本地图片路径必须存在。
- 本地图片路径必须放在 `/images/articles/` 下。
- 代码围栏数量必须成对。
- 表格分隔行必须包含 `---`。

输出风格：

- 通过时输出检查文章数量和成功提示。
- 失败时列出文件名和具体问题，方便网页新手定位。

## 正文样式设计

修改：

```text
src/styles/global.css
```

集中增强 `.prose`，不新建组件。

### 通用样式

`.prose` 负责 Markdown 正文：

- 控制最大宽度和行高。
- 设置段落间距。
- 优化链接样式。
- 设置 `h2`、`h3`、`h4` 层级。
- 设置列表缩进和列表间距。
- 设置引用块。
- 设置图片。
- 设置表格。
- 设置代码块。
- 设置分割线。

### Cyber 主题

Cyber 保持暗色霓虹，但正文区域不应该过度发光。

风格原则：

- 链接使用青色/粉色强调。
- 引用块可以有细霓虹边线。
- 代码块保留暗色终端感。
- 表格边线保持低亮度。
- 图片边框可以带轻微青色边缘。

### Anime-scroll 主题

Anime-scroll 使用纸张、墨线和印章感。

风格原则：

- 链接使用橙色或深蓝，不用霓虹阴影。
- 引用块像纸上批注，使用左侧墨线或橙色印记。
- 表格像手账表格，边框轻一些。
- 图片使用纸张边框和低强度投影。
- 代码块仍保持可读性，但减少赛博风视觉。

### 移动端

移动端重点：

- 表格允许横向滚动。
- 图片不超出容器。
- 代码块横向滚动。
- 标题不要因为太大挤压正文。

## 文章模板设计

新增：

```text
docs/templates/post-template.md
```

模板包含：

- 标准 frontmatter。
- 写作提醒注释。
- 三种文章结构提示：技术教程、经验复盘、感想随笔。
- 图片路径示例。
- 参考链接示例。

模板不放进 `src/content/posts/`，避免被 Astro 当成正式文章构建。

## 文档设计

新增：

```text
docs/内容规范.md
```

文档面向站主，不面向框架开发者。

内容包括：

- 每次写文章的流程。
- 文件命名规则。
- frontmatter 解释。
- 标签池。
- Markdown 标题规则。
- 图片规则。
- 三类文章推荐结构。
- 发布前自查清单。
- 常见错误和修正方式。

文档语言保持清楚、具体，不写抽象术语。

## 测试与验证

实现完成后运行：

```powershell
npm run verify:content
npm run verify:ux
npm run verify:motion
npm run verify:rss
```

如果正文样式或脚本影响构建，再运行 Astro build。由于当前系统默认 Node 可能不是 Astro 所需版本，构建时使用项目已知可用 Node：

```powershell
& 'C:\Users\刘\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\bin\node.exe' .\node_modules\astro\bin\astro.mjs build
```

构建后抽查：

- 文章详情页包含 `.prose` 样式。
- 长文图片、表格、引用块、代码块在两个主题下都有对应样式。
- `verify:content` 能指出内容错误，而不是只报模糊失败。

## 风险与处理

### 风险：检查脚本过严导致写文章负担变重

处理：

- 初版只检查低级错误和稳定规则。
- 摘要长度使用较宽范围。
- 标签池先保持小而清晰，后续可扩展。

### 风险：正文样式影响已有页面

处理：

- 只增强 `.prose`，它目前主要用于文章正文。
- 不改首页卡片、导航、主题切换控件。
- 保留两个主题已有视觉方向。

### 风险：现有文章不符合新规则

处理：

- 优先让脚本规则适配合理现状。
- 只对明显问题做小修正，例如图片路径检查或标题层级。
- 不在本次批量重写文章内容。

## 验收标准

这次任务完成时应满足：

- `docs/内容规范.md` 存在且可读。

- `scripts/verify-content.mjs` 存在，并能检查全部现有文章。

- `npm run verify:ux` 通过。

- `npm run verify:rss` 通过。
